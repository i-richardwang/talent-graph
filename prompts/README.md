# prompts/ — 任务说明文件库

存放派给 Agent 执行任务时的具体任务说明。每份 prompt 是一份完整的任务说明书,Agent 读完就能直接开始干活——不需要再去翻 `skills/` 才知道这次要做什么。

## 与 skills/ 的分工

| | `skills/` | `prompts/` |
|---|---|---|
| **是什么** | 方法 / 工具 | 任务 / 工单 |
| **回答什么** | 怎么做一个原子操作(method-level,domain-agnostic) | 这次要干什么、做完什么样,domain 业务定义 |
| **谁读** | 实际执行任务的 Agent | 拉起整次任务的 Agent / 调度方 |
| **复用粒度** | 项目级,跨任务、跨 domain 不变 | 任务级,绑定具体 domain / 批次 / 范围 |
| **更新频率** | 低(底层方法稳定后基本不动) | 高(每次任务、每次约定调整都可能新增 / 改) |

打个比方:SKILL 是工具说明书,prompt 是工单。Agent 拿到工单知道这次要做什么、做完什么样;中间用工具处理每个工作单元。

**关键分工**:SKILL 是 method-level、domain-agnostic 的——具体 domain 业务定义(如 school 的"挂母校名但不属于本校学历教育的不归属"、company 的"B 类只接受拆分独立上市子集团")由 prompt 给出。SKILL 不预设 domain 边界,谁有这个 domain 的任务谁负责给定义。

## 一份合格的 prompt 自检

写完一份新 prompt,从四个维度回头扫一遍——少答任何一条,Agent 拿到大概率会卡住:

1. **任务**:这是什么类型的工作?为什么做?
2. **输入**:具体在处理什么数据——一条原始写法?一份候选清单?一组员工?
3. **怎么做**:调用哪个 skill?有没有项目级的约束(domain 业务定义 / 白名单 / 范围限定 / 阈值)?
4. **完成标准**:怎么算做完了?要输出什么?

注意——这四条是要覆盖的维度,不是必须照搬的章节标题。把它们自然串成一份对 Agent 直说的工单即可,不用机械切成四节。

写完后再做一遍"陌生人测试":把自己当成完全没接触过这个项目的人,这份 prompt 是否独立可读、能否照着直接干。如果读完还要翻 SKILL 才搞清楚要做什么,这份 prompt 还没写完。

## 用法

调度方(人,或 `/orchestrate-tagging` 这类自动化编排程序)启动一次 Agent 任务时,把对应 prompt 作为任务说明喂给 Agent:

```
<prompt 文件内容>

(然后 Agent 自己按 prompt 往下推进,过程中需要时启动对应的 skills/)
```

批量场景由调度方把同一份 prompt 拆成多个 Agent 并发任务,每个任务独立处理一条数据。

## 目录结构

按 skill 分子目录,每个 skill 下放多个场景的 prompt。**文件名按业务场景命名,2-4 个英文 word**(如 `school-tier-tag.md` / `notable-employer-tag.md` / `subsidiary-lookup.md`)。两个反模式要避免:

- **机械按 entity domain 切**(`school.md` / `company.md`):看名字看不出业务场景,且不是每个 skill 都对每个 domain 都成立——如 `gather-entity-aliases` 业务上只适用于 school(学校 entity 边界清晰可批量筛),`attribute-raw-name` 业务上只适用于 company(零散主体逐条追溯股权)。机械按 domain 填会留下"不该存在的格子"。
- **用 `default` 这类技术语意词**:调度方一看不知道这次该用哪份。

```
prompts/
  attribute-raw-name/
    subsidiary-lookup.md
  define-tag/
    school-tier-tag.md
    notable-employer-tag.md
  gather-entity-aliases/
    school-aliases.md
  tag-employee/
    skill-judgment.md
    experience-judgment.md
```

新增 skill 或场景时按需建子目录。

## 现有 prompt 清单

### `attribute-raw-name/subsidiary-lookup.md`

- **业务场景**:招聘记录、员工简历、批量数据导入里不断冒出新的公司原始写法(比如 `字节跳动(上海)`、`菜鸟网络`、`某基金管理有限公司`),需要把它们一条条对应到我们维护的标准公司库。没对应上的原始写法,员工的这段工作经历就关联不到任何公司类标签——员工到底算不算阿里系、有没有互联网行业经历,这些标签判断会直接失真。
- **任务**:让 Agent 拿到一条原始公司写法,判断它在标准公司库里对应哪一家——可能是已有公司的另一种写法(比如 `ByteDance Inc.` 也是字节跳动);可能是值得单独拎出来挂在母公司底下的独立子品牌(比如菜鸟之于阿里);也可能没把握,这条就跳过不动它。一次 Agent 任务处理一条;批量场景由调度方拆成多个并发任务各跑一条。
- **本轮范围**:"单独拎出来做独立公司"的标准是"母公司已经/在拆分独立上市的子集团"(典型:菜鸟、蚂蚁、淘天);仅产品独立、运营独立不够(典型反例:飞书、企业微信、TikTok)。

### `define-tag/school-tier-tag.md`

- **业务场景**:人才图谱标签清单里"国内院校层次"和"海外院校层次"两类标签(藤校 / G5 / 985 / C9 / 双一流 / QS 前 100 …)需要逐个落到 talent-graph 里——把 tag 字段建好、再把对应的标准学校实体清单挂到 tag 下,下游 SQL 才能 JOIN 出这些标签的命中员工。
- **任务**:让 Agent 接到一个学校域名单标签名,判断这个清单是否成立(教育部官方名单、业内公认 → 直接做;多源口径冲突、动态榜单未锁年份、概念边界模糊 → 非零退出等消歧),清单成立时把应挂未挂的实体加挂、不再属于的解绑。一次 Agent 任务处理一个标签;批量场景由调度方拆成多个并发任务各跑一个。
- **本轮范围**:动态榜单类标签(QS / 泰晤士前 100 等)`[消歧说明]` 必须锁年份,否则视为清单不成立;`code` 也带年份后缀(如 `qs_top100_2025`),避免历史 tag 含义随榜单漂移。

### `define-tag/notable-employer-tag.md`

- **业务场景**:人才图谱标签清单里"知名机构经历"切面下的公司类标签(MBB / 四大 / 三中一华 / 红圈所 / BAT / TMD / FAANG …),以及"头部 PE/VC、互联网大厂、独角兽"等概念型标签,需要逐个落到 talent-graph 里——把 tag 字段建好、再把对应的标准公司实体清单挂到 tag 下,下游 SQL 才能 JOIN 出这些标签的命中员工。
- **任务**:让 Agent 接到一个公司域名单标签名,先按项目共识判这个清单是不是"业内公认 + 数量稀少 + 边界清晰"——是就直接 WebSearch 核实后挂上,不是就要看任务参数里有没有给消歧依据,没给则非零退出等消歧。一次 Agent 任务处理一个标签;批量场景由调度方拆成多个并发任务各跑一个。
- **本轮范围**:含"头部 / 大厂 / 巨头 / 独角兽 / 待补 / 等"字样的标签都视为模糊概念,任务参数 `[消歧说明]` 必须给定以下任一才能直接做——具体清单 / 权威来源+版本 / 可量化口径。不靠 Agent 凭常识从多份榜单里挑一份。

### `gather-entity-aliases/school-aliases.md`

- **业务场景**:talent-graph 已注册一批标准学校实体(清华大学、北京大学等),员工简历 / 招聘数据里这些学校的写法五花八门(`THU` / `清大` / `五道口金融学院` / `清华(深圳)` …),需要把这些写法登记成 alias,下游 SQL JOIN 才能识别出"这段教育经历属于这所学校"。
- **任务**:让 Agent 拿到一个标准学校实体 + 一片候选 raw_name 清单(CSV),通读全片,把真正属于该校的写法登记成 alias,其余跳过。一次 Agent 任务处理一个目标实体;批量场景由调度方拆成多个并发任务各跑一个目标实体。
- **本轮范围**:talent-graph 的学校 entity = 学历教育归属——挂母校名但不属于本校学历教育的主体一律跳过(附中 / 附属医院 / 园区 / 控股关联 / 培训机构等)。下属研究院 / 学院 / 改名前身 / 字符变体算归属。

### `tag-employee/skill-judgment.md` 与 `tag-employee/experience-judgment.md`

判定标签(`mode='assertion'`)按 `kind` 分两类,各用一份 prompt——两类的判定证据口径不同,不要混批:

- **`skill-judgment.md`(`kind='skill'`,技能 / 方法论)**:推荐排序 / 因果分析 / 实验设计 / 特征工程 / 模型训练上线 等。证据锚在工作经历正文里"具体用这套方法做了事"的段落,岗位名 / 团队名不算。
- **`experience-judgment.md`(`kind='experience'`,业务情境阅历)**:0→1 搭业务 / 国际化出海 / 商业化变现 / 高速增长期 / 跨部门大型项目 等。证据锚在业务的阶段 / 性质 / 范围,"在国际化公司"不等于"做过国际化业务"。

- **业务场景**:这两类都是判定标签——没有具名实体清单,要 LLM 通读员工 profile(教育 + 工作经历 + 简历原文)按 description 边界 prose 综合判决。
- **任务**:让 Agent 拿到一个员工 + 一组同 `kind` 的判定 tag,通读 profile 后对每个 tag 判是否命中,hit 写 `employee_tag_map`,miss / 证据不足跳过。一次 Agent 任务处理一个员工 + 多个 tag(profile 一次拉,多 tag 复用);批量场景由调度方拆成多个并发任务各跑一个员工。
- **domain 举例**:每份 prompt 列本域常见的模糊边界(技能类如"因果分析"算不算 AB 报表、经验类如"0→1"含不含内部孵化),帮 Agent 校准本域的临界形态。"边界吃不准就跳过并点名、让业务方迭代 description"这条判决方法是跨 domain 通用的,落在 `tag-employee/SKILL.md`,prompt 不复述。
